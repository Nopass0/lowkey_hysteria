# Архитектура Сервера Lowkey VPN (Go)

Серверная часть Lowkey VPN написана на **Go** с использованием протокола **QUIC** (через библиотеку `quic-go`) и интеграцией с Базой Данных проекта (PostgreSQL/Prisma) и брокером сообщений (Redis).

## Основные Компоненты

### 1. Серверная Шифрованная Транзитная Связь (QUIC)

Мы используем протокол QUIC поверх UDP для передачи данных.
В отличие от классического TCP-базированного OpenVPN или Shadowsocks, QUIC (который лежит в основе спецификации Hysteria2) предоставляет:

- **Мультиплексирование:** Данные могут идти паралелльно по разным стримам или дейтаграммам.
- **Дейтаграммы (Datagrams):** Ненадежная, но максимально быстрая доставка пакетов, идеальная для туннелирования сырого IP трафика.
- **Встроенное шифрование:** TLS 1.3 обязательно встраивается в сам handshake (рукопожатие) протокола QUIC. Мы генерируем самоподписанный ECDSA сертификат (P-256) прямо в оперативной памяти при старте сервера.

### 2. Интерфейс TUN (водопровод)

Мы используем библиотеку `water` для создания виртуального сетевого интерфейса `tun0` в Linux `(172.20.0.1/24)`.

- `tun0` принимает в себя «голые» IP пакеты.
- Мы включаем маршрутизацию (IP Forwarding) в ядре Linux (`sysctl -w net.ipv4.ip_forward=1`) и MASQUERADE в `iptables`. Это означает, что сервер работает как домашний роутер: подменяет IP-адрес отправителя из подсети `172.20.0.0/24` на свой внешний публичный IP и отправляет запрос в большой интернет.

### 3. База Данных и Авторизация

- **Подключение:** Сервер напрямую подключается к PostgreSQL.
- **Регистрация сервера:** При каждом запуске, сервер создает/обновляет о себе информацию в таблице `vpn_servers` (IP-адрес, порт, статус).
- **Рукопожатие (Handshake):** Клиент отправляет структуру `ClientHello` в формате MessagePack поверх двунаправленного QUIC стрима.
- **Валидация JWT:** Сервер проверяет переданный клиентом JWT токен, выданный основным сервером аутентификации сайта. Если токен валиден — сервер проверяет, есть ли у пользователя `isLifetime = true` или истекло ли время активной подписки (`activeUntil`).
- **Синхронизация:** Каждые 30 секунд сервер отправляет Heartbeat в БД для таблицы `vpn_servers`, обновляя поле `"lastSeenAt"`.

### 4. Инкапсуляция Данных (Пересылка)

После успешного рукопожатия (Handshake), сервер переходит в режим ожидания фреймов (Datagrams):

- **С Клиента на Сервер (`forwardQUICToTUN`):** `quic-go` получает зашифрованную дейтаграмму, дешифрует её и передает серверу как байтовые данные. Мы тут же записываем эти байты (сырой IP пакет) в устройство `tun0`. Ядро ОС Linux принимает пакет и роутит его во внешний интернет.
- **С Сервера на Клиент (`forwardTUNToQUIC`):** Ядро Linux получает ответ от сайта из интернета и направляет пакет в `tun0`. Наша программа в бесконечном цикле (1 цикл на сервер) читает `tun0`, определяет по Target IP пакету, какому именно Клиенту этот пакет адресован (через Mapping Map), и вызывает `SendDatagram` у конкретного подключения клиента по протоколу QUIC.

## Флоу подключения (Lifecycle)

1. **Запуск:** Сервер загружает ключи, настраивает `tun0`, прописывает `iptables` NAT, регистрируется в Prisma БД и начинает прослушку на UDP порту `:7000`.
2. **Accept:** Поступает UDP пакет. `quic-go` устанавливает защищенное соединение.
3. **Stream Handshake:** Открывается Stream, читается MessagePack авторизация, БД валидирует подписку. Клиенту назначается IP (Pool `172.20.0.X`) и отправляется ответный MessagePack (`ServerHello`).
4. **Data Transmission:** Запускается горутина чтения конкретно этого QUIC соединения. Любая отправленная клиентом дейтаграмма выбрасывается в локальный `tun0` Linux'a.
